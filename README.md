# Website Audit Tool

Internal Next.js tool for auditing a webpage, extracting factual metrics, and generating AI-driven improvement guidance.

## Getting Started

Install dependencies and run the app:

```bash
npm install
npm run dev
```

Open [http://localhost:3000](http://localhost:3000).

Environment variables:

```bash
GEMINI_API_KEY=your_key_here
```

## Architecture Overview

The system is organized around a simple audit pipeline:

1. The user enters a URL in the frontend.
2. The frontend opens an SSE connection to `GET /api/audit/stream?url=...`.
3. The backend fetches the page HTML once.
4. The backend parses the HTML with Cheerio.
5. The backend extracts deterministic metrics from the DOM.
6. Metrics are streamed to the client immediately.
7. The backend sends the metrics and page text to Gemini.
8. The backend validates Gemini output and streams the final result.

### Main Modules

- `src/app/page.tsx`
  Frontend page that starts the audit and listens to SSE progress/result events.

- `src/app/api/audit/stream/route.ts`
  Streaming endpoint that orchestrates the audit lifecycle and emits progress updates.

- `src/lib/scraper/fetchPage.ts`
  Responsible for downloading the target page HTML.

- `src/lib/scraper/extractDom.ts`
  Wraps Cheerio parsing and returns a DOM-like query API.

- `src/lib/metrics/computeBasicMetrics.ts`
  Extracts factual signals such as headings, link counts, image counts, alt coverage, and metadata.

- `src/lib/ai/geminiClient.ts`
  Creates the Gemini client/model from environment configuration.

- `src/lib/ai/generateInsights.ts`
  Builds the prompt, calls Gemini, parses JSON, and validates the returned structure.

- `src/components/AuditResult.tsx`
  Displays metrics, AI insights, and recommendations in a readable audit layout.

### Streaming Model

The current architecture uses Server-Sent Events instead of polling.

The backend emits:

- `progress`
- `metrics`
- `result`
- `failed`

This improves perceived performance because metrics appear before AI analysis completes, and the client no longer has to repeatedly call a status endpoint.

## AI Design Decisions

### 1. Deterministic data first, AI second

The system separates:

- factual metrics extracted from the DOM
- subjective analysis generated by Gemini

This keeps objective measurements stable and limits the AI's role to interpretation and prioritization.

### 2. Prompt constrains structure

Gemini is asked to return strict JSON with a known schema:

- `seo_analysis`
- `messaging_analysis`
- `cta_analysis`
- `content_depth`
- `ux_analysis`
- `recommendations`

This makes the response easier to validate and render in the UI.

### 3. Input is bounded

Page text is normalized and truncated before being sent to Gemini. This reduces token usage and keeps responses faster and more predictable.

### 4. Defensive parsing and validation

The AI response is not trusted blindly. The code:

- strips markdown fences if present
- parses JSON
- validates required sections
- clamps scores to `0-100`

This protects the UI from malformed model output.

### 5. Prompt injection mitigation

The prompt explicitly tells Gemini to treat page text as data, not instructions. This is a basic safeguard against prompt injection from the target webpage.

## Trade-offs

### SSE instead of polling

Pros:

- fewer repeated requests
- better perceived responsiveness
- cleaner progress UX

Cons:

- more stateful than plain request/response
- some hosting environments require care around long-lived connections

### In-memory processing

Pros:

- simple to implement
- no database or queue required
- fast local development

Cons:

- not durable across restarts
- not ideal for high concurrency or horizontally scaled workers

### Cheerio instead of a headless browser

Pros:

- lightweight
- fast
- enough for static HTML analysis

Cons:

- does not execute client-side JavaScript
- can miss content rendered only in the browser

### Single-model analysis flow

Pros:

- easy to reason about
- smaller implementation surface
- straightforward response schema

Cons:

- one model call handles multiple concerns at once
- less flexible than separate specialized prompts or multi-pass evaluation

### Heuristic metrics

Pros:

- cheap to compute
- easy to explain
- stable output

Cons:

- CTA detection is keyword-based and imperfect
- word count and link classification are useful approximations, not semantic truth

## What I Would Improve With More Time

### 1. Stronger scraping support

- Add optional headless browser rendering for JavaScript-heavy pages.
- Respect robots rules and add clearer fetch constraints.
- Add content-type and payload-size safeguards.

### 2. Better metric quality

- Extract main content instead of full-body text.
- Add canonical, robots, Open Graph, structured data, and title/meta length checks.
- Improve CTA detection with better semantic heuristics.

### 3. More robust AI layer

- Add retries and fallback behavior for malformed responses.
- Introduce prompt versioning and model-level evaluation.
- Consider separate passes for SEO, messaging, and UX for better control.

### 4. Production-grade job handling

- Replace in-memory flow with durable job storage or a queue.
- Add observability for latency, failures, and model costs.
- Add cancellation and timeouts for long-running audits.

### 5. Better frontend workflow

- Add copy/export actions for recommendations.
- Add section collapse/expand for long audits.
- Add historical audit comparisons and saved reports.

### 6. Testing

- Unit tests for metrics extraction
- Integration tests for streaming responses
- Contract tests for AI response validation

## Deployment Notes

The app is designed for Vercel deployment, but long-lived streaming behavior should be validated in the target runtime. If usage grows, the audit pipeline should move to a background job architecture with persistent storage.
